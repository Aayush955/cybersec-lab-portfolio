01
Environment Setup — KVM + virt-manager
Install and configure the KVM virtualization stack on the Linux host
KVMQEMUvirt-managerlibvirtOVMF
✔ DONE▼
Objective

Set up KVM (Kernel-based Virtual Machine) on a Linux laptop as the hypervisor for all lab VMs. KVM uses hardware-assisted virtualization (Intel VT-x or AMD-V) to run VMs at near-native CPU speed. We also install virt-manager as a GUI and libvirt as the virtualization management layer.

Commands
# Step 1 — Verify CPU supports hardware virtualization egrep -c '(vmx|svm)' /proc/cpuinfo sudo kvm-ok # Step 2 — Install full KVM stack sudo apt update && sudo apt install -y \ qemu-kvm libvirt-daemon-system libvirt-clients \ bridge-utils virt-manager virtinst ovmf # Step 3 — Add user to required groups (log out after) sudo usermod -aG libvirt,kvm $USER # Step 4 — Start and enable libvirt service sudo systemctl enable --now libvirtd sudo systemctl status libvirtd # Step 5 — Verify virsh list --all virt-manager
Common Issues & Solutions
⚠kvm-ok says "KVM acceleration can NOT be used"▼
Cause

CPU virtualization (VT-x/AMD-V) is disabled in BIOS/UEFI or unsupported.

Fix

Reboot → enter BIOS (F2/F10/Del during boot) → find "Intel Virtualization Technology" or "SVM Mode" → Enable → Save and exit.

⚠virt-manager: "Unable to connect to libvirt"▼
Cause

User not in libvirt group, or libvirtd not running.

Fix
sudo systemctl restart libvirtd sudo usermod -aG libvirt $USER # Log out and back in, then verify: groups $USER
⚠Permission denied running virsh commands▼
Fix
echo 'export LIBVIRT_DEFAULT_URI="qemu:///system"' >> ~/.bashrc source ~/.bashrc
Key Takeaway

KVM is a Type-1 hypervisor built into the Linux kernel — it runs VMs with near-native performance because it uses real CPU virtualization extensions. Unlike VirtualBox (Type-2), KVM passes CPU instructions directly to hardware. This is why enterprise data centers use KVM for production workloads.

02
Virtual Network Design & Segmentation
Create three isolated virtual networks using libvirt XML definitions
virshlibvirt XMLLinux bridgesNAT
✔ DONE▼
Objective

Create three logically separate virtual networks. LAN and DMZ are fully isolated — they have no direct path to the internet or each other. All traffic must pass through pfSense. This mirrors how real enterprise networks use VLANs to segment traffic and enforce policies at the firewall.

Commands
# Run the setup script bash scripts/setup-kvm-networking.sh # Verify all three networks running virsh net-list --all # Confirm bridges exist on host ip link show | grep virbr brctl show
Common Issues & Solutions
⚠Network already exists error▼
virsh net-destroy lab-lan virsh net-undefine lab-lan # Then re-run the script
⚠Networks don't survive reboot▼
virsh net-autostart lab-wan virsh net-autostart lab-lan virsh net-autostart lab-dmz sudo systemctl enable libvirtd
Key Takeaway

Isolated networks in libvirt have no default route — packets can't leave the segment without a router. The bridge interfaces (virbr-lan, virbr-dmz) are visible on the host, which means you can run tcpdump on the host to capture all lab traffic regardless of firewall state — a powerful debugging and analysis technique.

03
pfSense Deployment & Configuration
Deploy pfSense as the network gateway with WAN, LAN, and DMZ interfaces
pfSense CE 2.7DHCP ServerDNS ResolverNTP
⚡ IN PROGRESS▼
Objective

pfSense is a FreeBSD-based open-source firewall and router. It acts as the single gateway between all three network segments. Install pfSense as a VM, assign its three network interfaces, configure DHCP to hand out IPs to VMs, enable DNS resolver, and harden the default configuration.

VM Setup & Interface Assignment
# VM specs: 1024MB RAM, 1 vCPU, 8GB disk # Attach 3 NICs in virt-manager: # NIC 1: lab-wan → becomes em0 (WAN) # NIC 2: lab-lan → becomes em1 (LAN) # NIC 3: lab-dmz → becomes em2 (DMZ) # After install, pfSense console assigns: # WAN → em0 → DHCP → gets 192.168.100.x # LAN → em1 → static 10.0.1.1/24 # DMZ → em2 → static 10.0.2.1/24 # Access web UI from Kali (on LAN): # URL: http://10.0.1.1 User: admin Pass: pfsense # CHANGE THE PASSWORD IMMEDIATELY
Common Issues & Solutions
⚠pfSense WAN not getting DHCP from KVM NAT▼
Cause

KVM bridge needs promiscuous mode, or VM NIC is on wrong network.

Fix
sudo ip link set virbr-wan promisc on # Check pfSense got an IP via console virsh console pfsense-vm
⚠Can't access pfSense web UI at 10.0.1.1▼
Fix
# On Kali — check if you have 10.0.1.x address ip addr show # If not, request DHCP manually sudo dhclient eth0 ping 10.0.1.1
04
Deploy Internal VMs — Kali Linux & Ubuntu Server
Create attacker and target machines on their respective network segments
Kali Linux 2024Ubuntu Server 22.04SSHApache2
PLANNED▼
Objective

Deploy Kali Linux as the attacker on the LAN segment and Ubuntu Server as the target in the DMZ. Configure both with static IPs. Install vulnerable target services on Ubuntu (Apache, SSH, vsftpd) to give Kali realistic attack surfaces.

Commands
# On Ubuntu Server — set static IP via netplan sudo nano /etc/netplan/00-installer-config.yaml # Set: address 10.0.2.10/24, gateway4: 10.0.2.1 sudo netplan apply # Install target services sudo apt install -y apache2 openssh-server vsftpd sudo systemctl enable --now apache2 ssh # From Kali — verify connectivity ping 10.0.2.10 curl http://10.0.2.10 ssh ubuntu@10.0.2.10
Common Issues & Solutions
⚠Kali can't reach Ubuntu even though both are running▼
Cause

pfSense default rules block LAN to DMZ traffic. This is expected — correct behavior until Phase 5.

Fix

Complete Phase 5 firewall rules first. To test temporarily: pfSense web UI → Firewall → Rules → LAN → add a temporary allow-all rule. Remove after testing.

Common Issues & Solutions
⚠Rules applied but traffic still blocked/allowed unexpectedly▼
Cause

pfSense evaluates rules top-to-bottom and stops at the first match. Rule order matters — more specific rules must go above general ones.

Fix

Use Diagnostics → Packet Capture in pfSense to see what traffic is hitting each interface. Use Diagnostics → States to see what connection states exist. Re-order rules so specific rules appear above broad ones.

Network Analysis — Nmap & Wireshark
Perform reconnaissance and capture traffic for deep analysis
NmapWiresharktcpdumpNetcat
PLANNED▼
Key Nmap Scans
# Host discovery nmap -sn 10.0.2.0/24 # SYN scan — fast, stealthy nmap -sS -T4 10.0.2.10 # Full scan — all ports + service versions + OS detection nmap -sS -sV -O -p- --open 10.0.2.10 \ -oN reports/nmap-full.txt # Vulnerability scan via NSE scripts nmap --script vuln 10.0.2.10 \ -oN reports/nmap-vuln.txt # Capture on host bridge while scanning sudo tcpdump -i virbr-dmz -w captures/scan.pcap
Common Issues & Solutions
⚠Nmap shows all ports filtered — nothing visible▼
Cause

pfSense firewall is blocking the scan. LAN→DMZ rules haven't been configured yet.

Fix

Complete Phase 5 first and add a LAN→DMZ allow rule for scanning. Or capture on the host bridge (virbr-dmz) with tcpdump to confirm packets are arriving regardless of firewall rules.

Common Issues & Solutions
⚠Suricata running but no alerts during Nmap scan▼
Cause

Suricata is monitoring the wrong interface, or HOME_NET is not set to your lab subnets.

Fix
sudo nano /etc/suricata/suricata.yaml # Set HOME_NET: "[10.0.0.0/8,192.168.0.0/16]" # Set interface to your VM's eth0 or ens3 sudo systemctl restart suricata

Configuration
# pfSense web UI → VPN → WireGuard → Tunnels → Add # Listen Port: 51820 # Tunnel Address: 10.10.0.1/24 # Client config (Android WireGuard app): # Address = 10.10.0.2/24 # DNS = 10.0.1.1 # Endpoint = [WAN IP]:51820 # AllowedIPs = 10.0.1.0/24, 10.0.2.0/24
Common Issues & Solutions
⚠VPN connects but can't reach LAN/DMZ hosts▼
Cause

pfSense firewall has no rule allowing VPN tunnel traffic into LAN/DMZ segments.

Fix

In pfSense → Firewall → Rules → WireGuard interface → Add rule allowing traffic from VPN subnet (10.10.0.0/24) to LAN and DMZ.

06 — GitHub

GitHub Setup Guide

Two complete paths — standard Linux/Mac and the Android Termux method used to build this portfolio.

1

Install Git

Git is the version control system that powers GitHub. Install it on your machine first.

# Ubuntu / Debian / Kali sudo apt update && sudo apt install -y git # macOS brew install git git --version
2

Configure Git Identity

Git attaches your name and email to every commit. Set this once globally.

git config --global user.name "Your Full Name" git config --global user.email "your@email.com" git config --global init.defaultBranch main git config --global color.ui auto git config --global --list
Privacy tip: GitHub Settings → Emails → enable "Keep my email private" and use the noreply address shown.
3

Generate SSH Key

SSH keys replace passwords for pushing code. More secure and no typing credentials each time.

ssh-keygen -t ed25519 -C "your@email.com" eval "$(ssh-agent -s)" ssh-add ~/.ssh/id_ed25519 # Copy this entire output and add to GitHub cat ~/.ssh/id_ed25519.pub
Then: GitHub → Settings → SSH and GPG keys → New SSH key → paste the output above.
4

Install GitHub CLI

curl -fsSL https://cli.github.com/packages/githubcli-archive-keyring.gpg \ | sudo dd of=/usr/share/keyrings/githubcli-archive-keyring.gpg echo "deb [arch=$(dpkg --print-architecture) \ signed-by=/usr/share/keyrings/githubcli-archive-keyring.gpg] \ https://cli.github.com/packages stable main" \ | sudo tee /etc/apt/sources.list.d/github-cli.list > /dev/null sudo apt update && sudo apt install -y gh gh auth login
5

Create & Push Portfolio Repo

mkdir cybersec-lab-portfolio && cd cybersec-lab-portfolio git init echo "# Cybersecurity Lab Portfolio" > README.md git add . && git commit -m "Initial commit" gh repo create cybersec-lab-portfolio \ --public \ --description "Enterprise network security home lab" \ --source=. --remote=origin --push
6

Create Profile README

Create a repo named exactly as your username — GitHub displays its README on your profile page.

mkdir ~/YOUR-USERNAME && cd ~/YOUR-USERNAME git init nano README.md git add . && git commit -m "Add profile README" gh repo create YOUR-USERNAME --public \ --source=. --remote=origin --push
